这个博客的浏览器测试有 491 行，而它要保护的只是十来个静态页面。单看比例，很难说这是一笔划算的买卖。

Eleventy 已经能告诉我模板有没有成功生成，链接检查也能找出不存在的资源。可这两件事都无法回答一个更普通的问题：页面在 320 像素宽的手机上，究竟有没有东西挤出屏幕？

## 两轮检查各查一类错误

构建后的第一轮检查不打开浏览器。它把 `_site` 当成一张文件关系图：页面有没有唯一标题和描述，canonical 是否指向正确地址，本地链接能不能落到真实文件，搜索索引、订阅源、站点地图和 robots.txt 是否彼此一致。

这类检查速度快，而且错误指向明确。一个 Atom 条目链接错了，不必启动 Chrome 才能发现。

但它看不见布局，也不知道按钮点下去以后发生了什么。第二轮才启动真实 Chrome，通过 DevTools Protocol 发送导航、调整视口、执行脚本和截取画面。布局和交互留给这一轮，链接与元数据不必跟着进浏览器。

## 不做整页截图比较

脚本会在 320、375、768 和 1440 四种宽度打开首页，也会留下截图。但截图只是失败时的线索，不是通过与否的标准。

像素快照对字体渲染、操作系统和浏览器版本太敏感。一个字形的抗锯齿变化就可能让整张图变红，而真正重要的错误——例如导航刚好压住标题——反而淹没在差异里。

所以测试读取元素的几何信息，并断言几个不会轻易过时的关系：

- 文档宽度不能大于视口宽度；
- 首页标题不能越过左右边缘；
- 可见的页头控件必须留在视口内；
- 品牌、导航和右侧工具彼此不能相交。

这些断言不关心标题距离顶部是 97 还是 101 像素，却能抓住横向滚动条和元素重叠。测试的是布局意图，不是一次偶然渲染出的全部像素。

320 是压力边界，375 接近常见手机，768 观察断点附近，1440 检查桌面布局。CSS 以后增加断点时，再调整这组数字。

## 主题测试必须区分“选择”和“跟随”

明暗主题有两条来源：用户自己保存的选择，以及操作系统偏好。它们看起来都只是在 `<html>` 上得到一个 `data-theme`，优先级却不同。

测试先在 `localStorage` 写入浅色，重新加载后点击切换按钮，确认 DOM 状态、保存值和浏览器主题色一起变成深色。然后它删除保存值，依次模拟系统浅色和深色，确认页面会跟随系统变化，同时不偷偷向 `localStorage` 写入一份“用户选择”。

如果只断言页面最后是深色，这个优先级错误不会暴露。

## 搜索测试关心的是焦点，不只是结果

搜索打开以后，脚本不只检查面板类名。它还确认输入框获得焦点、正文进入 `inert` 状态、触发按钮的 `aria-expanded` 已更新。输入关键词后，结果链接必须指向实际文章或短记锚点。

接着脚本派发 Escape 键，并确认焦点回到原来的搜索按钮，正文恢复交互。一个搜索框即使能返回正确结果，只要关闭后把键盘使用者丢在页面开头，就仍然是坏掉的。

## 我特意关掉了一次 JavaScript

脚本访问文章页后，通过 DevTools Protocol 禁用 JavaScript，再重新导航。此时文章标题必须存在，正文必须有实际内容，页面也要保留足够的普通链接供人继续浏览。

这个站的搜索、主题和背景音乐都依赖 JavaScript，文章标题、正文和普通链接不依赖。脚本只断言后一组内容仍然存在。

最后还有一个不存在的地址。404 页面必须带正确标题和 `noindex, nofollow`，控制台里除了预期的 404 资源错误，不能出现新的异常。

## 为什么没有直接装一个测试框架

这份脚本自己建立 WebSocket，给 Chrome 分配调试端口，等待页面、发送 CDP 命令，最后负责关闭浏览器和 Eleventy 服务。很多代码都在做 Playwright 或 Puppeteer 已经做得更好的事。

当前项目的开发依赖只有 Eleventy，直接使用系统里的 Chrome 可以把新增依赖保持为零，也让我准确控制需要的那一点能力。代价同样明显：等待条件、页面生命周期和错误收集都得自己维护。项目如果继续增加表单、多页面交互或网络模拟，换成成熟框架会比扩建这套小客户端更合理。

491 行不是测试充分的证据。能明确说出的只有当前覆盖范围：窄屏不横向滚动，主题尊重选择，搜索可以退出，没有 JavaScript 仍能读，错误地址不会冒充正常页面。
